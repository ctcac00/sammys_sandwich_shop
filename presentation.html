Snowflake Stored Procs vs dbt

Same transformation outcome, very different amount of framework overhead

Developers waste ~60% of their SP effort on plumbing that dbt handles automatically.
Boilerplate

Stored procedures require wrapper code and manual row handling.

Focus

dbt models stay focused on transformation SQL and business logic.

Testing gap

SP pipelines lack built-in tests, so teams maintain custom data-quality checks.

Run complexity

SP pipelines often need custom chaining and retries; dbt provides DAG-native execution.

CREATE OR REPLACE PROCEDURE sp_enrich_orders()
RETURNS VARCHAR
LANGUAGE SQL
AS $$
DECLARE
    v_rows_inserted INTEGER;
BEGIN
    TRUNCATE TABLE enriched_orders;

    INSERT INTO enriched_orders (
        order_id, customer_id, employee_id,
        location_id, order_date, order_time,
        order_type, subtotal, tax_amount,
        discount_amount, total_amount,
        payment_method, order_period,
        is_weekend, has_discount, has_tip,
        /* ... 8 more columns ... */
    )
    SELECT
        o.order_id,
        NULLIF(o.customer_id, 'NULL'),
        TRY_TO_DATE(o.order_date, 'YYYY-MM-DD'),
        TRY_TO_NUMBER(o.subtotal, 10, 2),
        CASE WHEN HOUR(...) BETWEEN 6 AND 10
             THEN 'Breakfast'
             WHEN ... THEN 'Lunch'
        END AS order_period,
        /* ... more transforms ... */
    FROM SAMMYS_RAW.orders o
    LEFT JOIN (...) ic ON o.order_id = ic.order_id
    WHERE o.order_id IS NOT NULL;

    v_rows_inserted := SQLROWCOUNT;
    RETURN 'SUCCESS: ' || v_rows_inserted;
END;
$$;

dbt Model

Databricks Notebooks vs dbt

Same transformation outcome, very different debugging workflow.

A 10-minute fix becomes a multi-hour investigation when errors surface 3 files from the root cause.
Failure signal

Notebook jobs often fail in a top runner script, not where the original bug first started.

Root cause path

Teams typically trace through %run chains and layer runners before finding the true issue.

Lineage view

dbt graphs show upstream and downstream dependencies directly, which makes impact analysis faster.

Time to fix

Notebook debugging can mean repeated context switching; dbt keeps the path model-centric.

dbt Model

Databricks DLT vs dbt

Quick takeaways from using DLT in practice

Lock-in compounds: each new pipeline raises the cost of ever moving off Databricks.
Neat

DLT lets teams define models and tests together in a single pipeline flow.

Tradeoff

Because it is Python-first, it can be less approachable for less technical users.

Constraint

DLT is Databricks-only, so it reduces platform flexibility.

Cost risk

Without careful configuration and run strategy, DLT can become less cost-efficient.

# dlt/silver/staging/stg_orders.py
import dlt
from pyspark.sql.functions import (
    col, trim, to_date, coalesce, lit
)
from pyspark.sql.types import DoubleType

@dlt.table(
    name="stg_orders",
    comment="Cleaned and typed order data",
    table_properties={"quality": "silver"}
)
@dlt.expect_or_drop(
    "valid_order_id",
    "order_id IS NOT NULL"
)
@dlt.expect(
    "valid_total", "total_amount >= 0"
)
def stg_orders():
    return (
        dlt.read("bronze_orders")
        .select(
            col("order_id"),
            trim(col("customer_id"))
                .alias("customer_id"),
            to_date(col("order_date"))
                .alias("order_date"),
            col("subtotal")
                .cast(DoubleType())
                .alias("subtotal"),
            # ... more columns ...
        )
    )

Key Takeaways

Same sandwich data, different operational tradeoffs.

  1. 1

    Code focus: dbt keeps models on business logic; native approaches add execution scaffolding.

  2. 2

    Debugging path: Native debugging is often multi-hop, so fixes take longer.

  3. 3

    DLT tradeoff: You get built-in expectations and pipeline flow, but it is Python-first and Databricks-specific.

  4. 4

    Decision criteria: Select based on team capabilities, governance needs, and long-term platform strategy.

If your priority is readable SQL plus cross-warehouse portability, dbt is usually the simplest default.
